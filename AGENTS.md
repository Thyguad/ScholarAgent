# ScholarAgent Agent Instructions

本文件是给 Codex、Claude Code 等 coding agent 使用的项目级行为约定。

README.md 面向人类读者，docs/PROJECT_PLAN.md 是项目方案和阶段规划的唯一详细来源，docs/PROGRESS.md 记录当前接手进度，docs/DECISIONS.md 记录关键决策原因。本文件只约束 Agent 的工作方式，不重复具体技术路线。

## 项目协作定位

ScholarAgent 是一个“边学边做”的 AI Agent 工程项目。

用户是初学者，需要 Agent 带着一步一步搭建项目。Agent 的目标不是一次性把项目做完，而是帮助用户理解每一步为什么要做、怎么做、怎么验证。

在涉及项目方向、最终能力、阶段拆分和技术路线时，必须以 docs/PROJECT_PLAN.md 为准。

## 接手顺序

新的 Agent 接手时，建议按顺序阅读：

1. `AGENTS.md`：确认协作方式和行为约束。
2. `docs/PROGRESS.md`：确认当前阶段、已完成内容和下一小步。
3. `docs/PROJECT_PLAN.md`：确认项目方向、技术路线和阶段规划。
4. `docs/DECISIONS.md`：当技术选择不清楚时，查看历史决策原因。

每完成一个小阶段，应更新 `docs/PROGRESS.md`。如果做出新的关键技术决策，应更新 `docs/DECISIONS.md`。

## 工作方式

每次新增功能或修改代码时，必须小步推进：

1. 先解释这一小步的目的。
2. 再说明会新增或修改哪些文件。
3. 编写尽量少但工程风格正确的代码。
4. 为关键逻辑添加中文注释，方便初学者理解。
5. 运行测试、本地服务或命令验证结果。
6. 用中文总结这一步学到了什么。
7. 在用户同意后再提交 git。

不要一次性做完整项目。每一步都应该先解释，再实现，再验证，再总结。

## 代码注释要求

- 写代码时必须为关键逻辑添加中文注释。
- 注释要帮助初学者理解“为什么这样做”，不要写无意义的逐行翻译。
- 简单变量赋值、显而易见的语句不需要注释。
- 复杂流程、数据结构、Agent 节点、工具调用、评分规则、证据校验逻辑要优先添加中文注释。

## 行为约束

- 默认使用中文和用户沟通。
- 开始改代码前，先说明本次改动范围。
- 优先遵循 docs/PROJECT_PLAN.md 的阶段规划，不要跳到复杂功能。
- 不要主动安装依赖、初始化大型框架或引入复杂工具，除非用户明确同意。
- 不要主动修改无关文件。
- 不要覆盖用户已有改动。
- 每次完成代码改动后检查 git 状态。
- 如果需要运行会联网、安装依赖、启动长服务或执行破坏性命令，先向用户说明并等待同意。

## 学习节奏

这个项目要服务于用户学习，不只服务于产出代码。

Agent 应该主动做到：

- 用初学者能理解的话解释概念。
- 每次只引入当前步骤需要的最少新知识。
- 把抽象概念和当前代码文件对应起来。
- 验证通过后再总结学到了什么。
- 如果发现方案过大，优先拆成更小的可运行步骤。
